# Impeccable Design Context

## Maintenance Rule

If frontend behavior or visual direction changes in a durable way, update this file in the same pass so the persistent design context stays aligned with the app.

## Design Context

### Users

Experienced Discogs collectors with 1,000+ release libraries who need to split, merge, or reorganize collections across accounts. They are technically comfortable, detail-oriented, and patient with complexity, but they are unforgiving of ambiguity during high-stakes operations.

**Job to be done:** execute collection migrations with confidence, knowing exactly what will move, where it will go, and what happened if something goes wrong.

### Brand Personality

**Bold. Technical. Focused.**

This is a serious power tool, not a consumer dashboard. Dense capability is acceptable. Vagueness, softness, and decorative noise are not.

### Emotional Goals

The interface should make users feel in control, informed, and safe performing consequential actions. Confidence matters more than friendliness.

### Current Visual Direction

- Exposed light-grid control surface with warm cream background and near-black ink
- TE-leaning orange accent, darker green success/source states, strong visible rules
- Square geometry throughout the interface
- IBM Plex Sans for dense UI copy and IBM Plex Serif for emphasis and numerals
- Discogs-adjacent editorial tone, not generic SaaS polish

### Anti-References

- No neon or cyberpunk effects
- No purple/cyan gradient language
- No soft, calm productivity-app beige minimalism
- No generic SaaS dashboard patterns
- No mobile-first consumer-app aesthetics

## Design Principles

1. **Density with clarity**: show a lot, but make the structure legible at a glance.
2. **Expose structure, not decoration**: borders, rules, alignment, and spacing should define the UI.
3. **Consequential actions deserve weight**: destructive flows need confirmation and danger treatment without theatrical alarmism.
4. **Keep capability close to its target**: source filters belong with the source table, not in distant control panels.
5. **Bounded dimensions should feel bounded**: enumerable filter values should be clickable options, not arbitrary free text.
6. **Accounts are setup, not the main workspace**: account controls belong with the Source and Destination sections they affect, not as a dominant rail surface or detached global dropdown. In those section headers, keep the order title first and account controls second. Put summary metadata in the toolbar row above the table. Show identity and sync freshness only; keep `Sync` primary and demote `Disconnect`. When a user needs to connect or replace an account, expand that setup inline inside the same section, lead with token entry, and keep browser OAuth secondary.
7. **Shell headers should stay structural**: anchor collapse and title on the left, then use the right side only for high-value controls or status that help the section read as one coherent unit.
8. **Review is the threshold step**: preview, blocking conflicts, and launch readiness should read as the main decision gate before execution. Lead with readiness, then show required fixes, then evidence.
9. **Operational noise is not product value**: server-side event chatter does not belong in the main job console. The console should read like an audit record with clear next actions and release-level outcomes.
10. **Theme can evolve, brand cannot**: future visual adjustments are fine only if they preserve precision, structure, and Discogs-world familiarity.
